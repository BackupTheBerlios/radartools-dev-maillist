Hello,
at the first glance it looks like it should be easy to do.

Probably it' s nothing special, but these are just a few points of
which I would think:
- good definition of plugin modules: i.e. the plugin procedure should
accept all the parameters (at least as phantoms) of usual modules,
like: /CALLED, /NO_GUI, /ARGUMENTS_STRUCTURE, ..., or something
similar. we need to define the common set of arguments, which should
be accepted by all modules.
- as well as a plugin-specific unnamed structure like
PLUGIN_INFO={menu_name="IDIOT", menu_position="insar...",...} , where
menu_position could be an array of positions.
- file name and location: e.g. in "rat/plugins/" and in
"~/.rat/plugins" , and rat could check this path automatically to load
all procedures which are named like e.g. ["rat_plugin_*.sav",
"rat_plugin_*.pro"]. The IDL name of the module function should then
also be the same as the file name without the suffix ".pro" or ".sav".
- everything else is only a technical problem of realization.
I would suggest to append the plugin modules at first at all
menu_positions as provided by the functions, as well as e.g. in the
menu "genera/plugins/", for the case that it was not possible to
append it correctly to menu_position.
Before constructing the RAT GUI, one has to check the plugin
directories for the modules, then adjust the menus, and then to
provide some framework to call these plugins.
That's almost all, or did I forget something?

It looks like an interesting task, I can perhaps also try to construct
something workable in the next days.
It's of course against the culture of RAT, but it would be nice to
have some definitions for module arguments, etc.

Tschüss,
Maxim
